Sector Research Pipeline

九阶段赛道研究管道 — 从数据预取到 Portal 发布，系统化产出投资级赛道报告。

核心理念

赛道研究的核心框架是三维度评估：TAM x 利润率 x 市占率。赛道的本质不是"这个行业有多大"，而是"谁能在这个行业里赚到钱、赚多少、能持续多久"。

河流模型是整个分析的骨架：

  • 河岸 Riverbank — 结构性信念，概率 >=90%，赛道级的长期判断。例如："支付网络效应不可逆"
  • 河流 River — 可证伪假设，每条假设绑定拐点和置信度评分。例如："稳定币跨境支付 2026 年占比突破 5%"
  • 分叉 Fork — 互斥路径，各路径概率合计 100%。例如："监管收紧 vs 沙盒开放"

赛道研究 vs 标的研究：赛道是标的的上游。赛道研究做的是横向对比 — 谁在打、打什么牌、牌桌规则是什么；标的研究做的是纵深 — 这个玩家的手牌、牌技和筹码够不够。先看懂牌桌，再挑玩家。

河流模型 — 认知架构

河流模型是整个 Pipeline 的结论骨架。不是"写完报告贴个标签"，而是从 S4 开始就用这个结构组织所有判断，Sentinel 每天用同一套结构做贝叶斯更新。一套框架贯穿建档 → 监控 → 更新 → 行动全链路。

三层结构

层级是什么概率语义失效 =State
河岸 Riverbank 不依赖其他假设的结构性判断（地基） 边际概率，独立成立
P ≥ 90 视为成立		 退出整个赛道 active / challenged / invalidated
河流 River 条件于河岸的方向性假设 条件概率
（河岸成立前提下）		 退出该主题全部仓位 active / challenged / invalidated

2-4 条互斥路径		 各路径条件概率

一句话区分：河岸是结构（几乎不变），河流是方向（随证据更新），分叉是决策节点（等事件 resolve 触发行动）。

河流分两类，都要有：

  • 方向性河流 — 跨层的投资判断，表达"我赌什么"（如"多轨化趋势让编排层侵蚀卡网络利润"）
  • 层级河流 — 某竞争层内的假设，表达"这个层会怎样"（如"稳定币重构跨境结算"）

DAG 依赖关系

河岸、河流、分叉构成有向无环图（DAG）：

riverbank A ──┐
              ├── river X ── fork X1 (paths: A/B/C)
riverbank B ──┘              fork X2 (paths: A/B)
              │
              └── river Y ── fork Y1

riverbank C ──── river Z ── fork Z1
  • river → 1+ riverbank(s) — 一条河流可以依赖多个河岸（交叉领域假设）
  • fork → 1 river — 一个分叉只挂一条河流
  • 一个河岸可以被多条河流引用 — 多对多关系

级联规则：

  • 河岸 P 跌破 90 → 依赖它的河流自动 challenged
  • 河流 invalidated → 其下所有 fork 自动 cancelled
  • 赛道分叉更新 → 通过 linked_symbols 传导到标的级 asset_judgment

实例：Global Payments DAG

GP-RB1 支付利润=基础设施租金 (p=83) ──┬── GP-H2 卡网吸收新rail (40) ── F1 Wero渗透
                                      ├── GP-H3 编排层拿权利 (40)   ── F1 PYPL多轨验证
                                      └── GP-H4 发行层租金压缩 (72)

GP-RB2 数字支付替代现金不可逆 (p=95) ──┬── GP-H1 稳定币重构跨境 (65) ── F1 GENIUS Act
                                       └── GP-H3 (共享，也依赖 RB1)

GP-RB3 跨境/B2B效率最低利润最厚 (p=93) ── GP-H1 (共享，也依赖 RB2)

置信度计算 — TP 驱动

所有概率由拐点（Tipping Points）驱动计算，不是拍脑袋。每个 TP 通过 hypothesis_id 绑定到具体的河岸、河流或分叉。一个假设可挂多个 TP，同假设下 TP 的 weight 合计 100。

河岸 / 河流 p_value

p_value = Σ(weight × score_map[score_status]) / Σ(weight) × 100
score_status分值含义
confirmed1.0已被事实验证
strong0.8强证据支持
approaching0.6接近验证
watching0.4观察中
weakening0.2证据转弱
failing0.0证据已反面

计算示例 — GP-RB1 "支付利润来自基础设施租金"：

  TP-1: Visa/MA 网络费占收入比   weight=40  score=confirmed  (1.0)
  TP-2: 监管是否限制 interchange  weight=30  score=watching   (0.4)
  TP-3: 替代轨道渗透率           weight=30  score=approaching (0.6)

  p_value = (40×1.0 + 30×0.4 + 30×0.6) / 100 × 100 = 70

  → 需要 TP-2 或 TP-3 走强（approaching→strong）才能回到 ≥90 河岸安全线

分叉路径概率 — path_signals + softmax

分叉的每条路径不只有概率，还预设了行动和预期影响。路径概率由 TP 的 path_signals 通过 softmax 驱动：

1. 每个 TP 携带 path_signals — 对各路径的方向性判断
   如：{"A": "利好", "B": "利空", "C": "中性"}

2. 信号五档映射 signal_weight：
   利好 = 1.0 ｜ 偏好 = 0.5 ｜ 中性 = 0 ｜ 偏空 = -0.5 ｜ 利空 = -1.0

3. 每条路径: raw = Σ(weight × score_map[score_status] × signal_weight)
4. 全部 raw 做 softmax 归一化 → paths.p，总和 = 100
5. 无 TP 的新 fork → paths.p 用人工初始值，有 TP 后自动接管

实例 — GP-H1-F1 GENIUS Act：

路径 A: 按期落地门槛适中  p=76%  → action: 加仓 PYPL         payoff: +10-15%
路径 B: 推迟 3-6 个月    p=10%  → action: 维持现有仓位       payoff: 中性
路径 C: 落地但门槛极高   p=14%  → action: 减仓 CRCL 加仓 V/MA payoff: CRCL -15-25%

新信号："参议院银行委员会 18:6 通过 GENIUS Act"
  → 归河: GP-H1-F1
  → 分类: 涨落（实质性立法进展）
  → TP score_status: watching → approaching
  → path_signals: {"A": "利好", "B": "偏空", "C": "中性"}
  → 重算 paths.p: A: 76→82, B: 10→7, C: 14→11

Sentinel 信号处理闭环

Pipeline 产出初始认知（S4 建档），Sentinel 每天用贝叶斯更新让认知持续迭代。同一套河流模型、同一套 TP 驱动公式，贯穿建档到日常监控。

信号分类

分类判定标准定义处理方式频率
浪花|Δp| ≤ 3短期噪声，不改变假设记录 signal_log，不更新 p_value最常见
涨落|Δp| ≤ 10有意义的证据变化记录 signal_log + 建议变档 → 等 Fred review每周数次
改道|Δp| > 10结构性变化，可能推翻河岸/河流记录 signal_log + 建议变档 + 告警 → 等 Fred review极少

三级分类是 signal_log 的审计标签（按模拟 Δp 幅度），不决定是否自动执行。所有 score_status 变档一律 needs_review。

完整更新流程

每日信号输入（新闻 / 财报 / 政策 / 链上数据）
    │
    ▼
  归河 — 匹配信号属于哪个 hypothesis_id
    │
    ▼
  分类 — 判断浪花 / 涨落 / 改道
    │
    ├─ 浪花 → 写 signal_log，结束（不改任何数值）
    │
    ├─ 涨落 → 写 signal_log（含 LR 交叉验证 + prior/posterior 模拟）
    │         → 建议 TP score_status 变档（如 watching → approaching）
    │         → 等 Fred review（所有变档一律 needs_review）
    │         → Fred approve 后：重算 p_value / paths.p → 级联传导
    │
    └─ 改道 → 写 signal_log + 推送 #daily 告警
              → 建议 state 变更 proposal（challenged / invalidated）
              → 等 Fred 确认后执行（河岸级必须人工确认）

似然比（LR）交叉验证

每次 Sentinel 建议变档（score_status 变化），额外评估证据的诊断性：

LR = P(看到这个证据 | 假设为真) / P(看到这个证据 | 假设为假)

  LR > 3.0    高诊断性 — 强力区分假设真/假
  LR 1.5-3.0  中等诊断性
  LR < 1.5    低诊断性（浪花） — 不管假设真假都可能看到

LR 由 Sentinel judgment 的 LLM 评估输出，不改变加权公式 — 它是交叉验证层，不是替代计算。交叉验证逻辑：比较 LR 和实际变档幅度，抓自相矛盾：

情况lr_check含义
LR > 3 但只动 1 档possibly_underestimated证据很强但变化保守，可能低估了影响
LR < 1.5 但动 ≥2 档possibly_overestimated证据弱但变化激进，可能过度反应
其他consistentLR 和变化幅度匹配

Signal_log 审计链

每次 score_status 变档都写入 signal_log.csv，记录完整决策链路：

信号 → 归河(hypothesis_id) → 变档(TP old→new score_status)
     → 模拟影响(prior_p → posterior_p, Δp)
     → LR 交叉验证(lr_check)
     → 决策(ai_proposed → human_accept/reject)
字段来源含义
hypothesis_id / tp_id10a 归河哪个假设的哪个拐点
old / new_score_statusjudgment 建议变档前后的 6 档状态
prior_p / posterior_pcalc_confidence_diff 模拟"如果这个 TP 变了，假设置信度从多少到多少"
likelihood_ratio / lr_checkjudgment + 交叉验证证据诊断性 + 是否和变档幅度一致
classificationΔp 幅度浪花(≤3) / 涨落(≤10) / 改道(>10)
decision_by流程状态ai_proposed → human_accept / human_reject

ID 规范：SL-TP-{date}-{seq}（TP 级变档）/ SL-HD-{date}-{seq}（hypothesis 级决策），避免同日跨文件碰撞。

Signal_log 解决的核心问题：追溯性。过去的判断为什么变了？变了多少？LLM 的理由是什么？Fred 同意了吗？没有这条日志，所有历史只有最终状态、没有中间过程。

Fork Resolve — 事件驱动行动

事件实际发生（不是概率到 95% 就提前 resolve）
    │
    ▼
  标记 resolved_path + resolved_at
    → 执行预设 action（仓位调整建议推送 Fred）
    → 写 events.csv 记录
    → 更新依赖该 fork 的 asset_judgment
    → 通过 linked_symbols 传导到标的级页面

防抖规则 — 5 条硬约束

避免对噪声过度反应：

  1. 浪花不动手 — 分类为浪花的信号只记录，不改 p_value
  2. 单一来源不改道 — 改道分类需要 2+ 独立来源交叉验证
  3. TP score 单次变动 ≤1 档 — 不允许从 watching 直接跳到 confirmed，必须经 approaching → strong
  4. 河岸 invalidated 需人工确认 — Sentinel 可以 propose challenged，但 invalidated 必须 Fred 确认
  5. Fork resolve 等事件实际发生 — 不因概率到 95% 就提前 resolve

盲区检测

"如果这条信息为真，会不会后悔没早建一个分叉？"
答案是"会" → 补建 fork，设定初始路径和概率。
答案是"不会" → 正常归河分类。

这是防止框架盲区的主动机制 — 不在框架内的信号最容易被忽略。

Sentinel judgment 每次运行都执行盲区扫描：检查当天证据中是否有不属于任何现有 TP的重要信号。发现后输出 unmatched_signals（含建议的新 TP 名称、类型、权重、理由），但新 TP 必须 Fred review 后才会被添加 — 不自动建 TP。

S0 数据预取

主 session 用 web_search + scanner-data 搜集赛道基础数据：市场规模、增长率、主要玩家、近期事件。

Agent：主 session（不 spawn）
输入：赛道名称 / Fred 的初始方向
输出：data/s0_prefetch.md
设计决策：每条数据必须附来源 URL，后续阶段可回溯验证。不做判断，只做收集。

S1 赛道画像

用 sector-intro agent 把原始数据组织成结构化赛道画像。

Agent：sector-intro（.claude/agents/sector/sector-intro.md）
输入：s0_prefetch.md
输出：knowledge/s1_landscape.md
设计决策：5 个模块缺一不可 — 赛道本质（一句话 + 第一性原理）、TAM 测算、商业模式拆解、产业运行机制、发展阶段判断。

S2 角色层识别

主 session 根据 S1 判断赛道的竞争层数，决定 S3 用简单模式还是多层模式。

Agent：主 session（不 spawn）
输入：s1_landscape.md
输出：分层方案（口头确认）
设计决策：这是第一个人工检查点。复杂赛道（如支付，4 层）和简单赛道（如 SaaS 细分，1-2 层）的研究深度差异很大，必须人工确认后再投入算力。

S3 竞争格局

按确认的分层方案，对每层做竞争格局分析。

Agent：sector-competition（.claude/agents/sector/sector-competition.md）
输入：s1_landscape.md + s0_prefetch.md
输出：knowledge/s3_competition*.md（多层模式下每层一个文件）
设计决策：重要玩家不跳过 — 分析深度由市场影响力决定，不由可投资性决定。非上市公司也要覆盖，否则竞争全景图是残缺的。

S4 综合判断

基于 S1-S3 全部材料，输出河流模型结论。

Agent：sector-judgment（.claude/agents/sector/sector-judgment.md）
输入：s1_landscape.md + s3_competition*.md + s0_prefetch.md
输出：knowledge/s4_judgment.md
设计决策：这是第二个人工检查点。输出河岸 / 河流 / 分叉 + 拐点清单 + 标的排序建议。Fred 在这里确认哪些标的进入 S5 深研，避免全量研究浪费算力。

S5 标的深研

对 Fred 确认的 top 标的，并行 spawn 深度研究。

Agent：stock-lead / crypto-lead（slim mode）
输入：s4_judgment.md + Fred 确认的标的列表
输出：各标的独立目录（3-universe/stock/{SYMBOL}/）
设计决策：slim mode 跳过重复的赛道背景，聚焦标的自身。重要但不可投资的玩家也做参考研究，确保竞争对比完整。

S6 完整报告

Merge S1-S5 全部材料为一份完整赛道报告。

Agent：sector-research（.claude/agents/sector/sector-research.md）
输入：s1 + s3 + s4 + s5 全部产物
输出：s6_report-YYYY-MM-DD.md
设计决策：Merge not summarize — 输出信息密度 >= 输入，可删的只有重复定义和过程性语言。核心独立价值在第七章（赛道级投资结论 + 标的排序 + 监控框架）。

S7 Challenge

独立 agent 对 S6 报告做系统性挑战。

Agent：sector-challenge（.claude/agents/sector/sector-challenge.md）
输入：s6_report.md + s3_competition*.md + s4_judgment.md
输出：knowledge/s7_challenge.md
设计决策：写报告的人不能审自己。7 个维度逐一检查，Score 1-10，>= 4 才通过。挑战结果反馈回 S6 做修正。

S8 建档

将最终报告转化为 Cortex 标准产物，发布到 Portal。

Agent：主 session（不 spawn）
输入：修正后的 s6_report.md
输出：CSV 文件 + page.md + Portal 页面
设计决策：11 项 checklist 逐项验证：hypotheses.csv / tipping_points.csv / events.csv / page.md YAML / render-pages / build.py / Portal 4 tab / notes.md / 标的 cross-link 等。

设计原则

六条核心原则，从实际研究中提炼：

  1. 结论层按收入池比例 — 不被单一子赛道叙事热度带偏。一个子赛道的分析篇幅应与其在整体 TAM 中的收入占比匹配，而不是与媒体关注度匹配。
  2. 重要玩家不跳过 — 分析深度由市场影响力决定，不由可投资性决定。Stripe 不上市，但不分析 Stripe 的支付赛道研究是残缺的。
  3. Merge not summarize — 输出信息密度 >= 输入，可删的只有重复定义和过程性语言。每一轮 agent 产出都是增量，不是摘要。
  4. Schema 合同 — sector-page-schema.md 是 page.md 和模板的唯一合同。schema 变了，page 和模板一起变；schema 没变，两边都不准自由发挥。
  5. 写报告的人不能审自己 — S7 用独立 challenge agent，和 S6 的 sector-research agent 完全隔离。自审没有信息增量。
  6. 证据分级 — [A] 一手数据（财报、官方披露）附 URL；[B] 研报 / 分析师估计应附 URL；[C] 推理和估算标注逻辑链。读者一眼看出哪些是事实、哪些是判断。

质量控制

S7 Challenge 是整个 pipeline 的质量关卡。独立 agent 从 7 个维度逐一检查 S6 报告：

  1. 赛道论点 — TAM 测算是否合理？增长假设有没有过度外推？
  2. 标的选择 — 有没有遗漏重要玩家？排序逻辑是否自洽？
  3. 估值假设 — 关键参数（增长率、利润率、折现率）是否在合理区间？
  4. 风险评估 — 尾部风险有没有被低估？黑天鹅场景覆盖了吗？
  5. 修正逻辑 — 如果核心假设错了，报告的结论会怎么变？
  6. 证据质量 — [A]/[B]/[C] 分级是否准确？关键结论有没有只靠 [C] 级证据？
  7. 竞争覆盖完整性 — 每一层的 top 3 玩家是否都有充分分析？

评分制度：Score 1-10。

  • Score >= 4 — 修正 challenge 指出的问题后建档
  • Score 1-3 — 停止流程，等人工判断是否需要重做某个阶段

历史案例：Global Payments 赛道 S7 得分 7.5/10，共 8 处修正（3 处估值参数调整、2 处竞争覆盖补充、2 处证据降级、1 处风险场景增补）。

与 Sentinel 的联动

Pipeline 产出初始认知（S4 建档），Sentinel 负责持续迭代。完整的信号处理流程、置信度重算机制和防抖规则见上方「河流模型 — 认知架构」章节。

  • 同一套数据 — Pipeline 写入的 hypotheses.csv / tipping_points.csv 就是 Sentinel 每天读取和更新的对象
  • 同一套公式 — TP 驱动的 p_value 计算（加权平均）和 paths.p 计算（softmax）在建档和日常更新中完全一致
  • 周期性 Review — 关键日期（财报、监管 deadline、行业大会）记入 events.csv，Sentinel 在日期前后主动拉取最新信息
  • 赛道→标的传导 — 赛道分叉更新 → linked_symbols → 标的级 asset_judgment 联动
